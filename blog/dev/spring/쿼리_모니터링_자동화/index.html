쿼리 모니터링 자동화

2025년 12월 29일 · 약 3분
thunder
개발자

인사모듈 SaaS를 만들면서 N+1 문제가 있다는 건 알았지만, 처음엔 Postman으로 몇 번 호출해 보고 로그를 훑는 정도로만 확인했다.

“쿼리 많이 나가네” 정도의 감은 있었지만, 이 방식으로는 지금 고친 게 나중에도 유지되는지 확신할 수가 없었다.

그래서 요청 한 번에 실제로 쿼리가 몇 개 나갔는지를 숫자로 남겨보자는 생각을 하게 됐다.
구조나 아이디어는 Codex 도움을 받아 뼈대를 잡고, 내 프로젝트 상황에 맞게 하나씩 수정하면서 구현했다.

내가 겪었던 한계​

  • API 호출 후 로그를 보며 감으로 판단
  • 개선 전/후를 숫자로 비교하기 어려움
  • 리팩토링 이후 N+1 재발을 막을 장치가 없음

그래서 해본 것(목표)​

  • 요청 단위로 쿼리 수 / 총 시간(ms) / 평균 시간(ms)을 수집
  • 로컬: 응답 헤더로 바로 확인
  • 테스트: 리포트를 자동으로 남기기 + 쿼리 한도(상한) 테스트로 회귀 방지

1) 요청 기준으로 숫자 남기기​

datasource-proxy의 QueryCountHolder로 요청이 처리되는 동안 실행된 JDBC 쿼리를 모았다.
그리고 Filter에서 응답 커밋 전에 헤더를 넣는 방식으로 정리했다.

이제 응답에서 바로 아래 값을 확인할 수 있다.

  • X-Query-Count
  • X-Query-Time-Ms
  • X-Query-Avg-Time-Ms

이제 “N+1 같아 보인다”가 아니라, 요청 한 번에 쿼리 몇 개인지 바로 볼 수 있었다.

2) 실제로 잡힌 N+1 예시: /api/attendance/me​

DTO 변환 과정에서 LAZY 연관을 건드리면서 N+1이 발생하고 있었다.
Repository에 @EntityGraph를 적용해 필요한 연관을 같이 로딩했다.

// before
List<Attendance> findByUserId(Long userId);

// after
@EntityGraph(attributePaths = {"user", "company"})
List<Attendance> findByUserId(Long userId);

헤더 기준 결과는 단순했다: 쿼리 수 3 → 1

3) 테스트로 남기기​

“로컬에서만 보고 끝나면 또 잊는다”는 생각이 들어 테스트에서도 같은 지표를 수집했다.

  • MockMvc로 주요 API 호출
  • 응답 헤더 수집
  • JSON 리포트 생성: build/reports/query-metrics/report.json

리포트는 아래 테스트를 실행하면 생성된다:

GRADLE_USER_HOME=$PWD/.gradle-home ./gradlew test --tests com.ji.ess.metrics.QueryMetricsReportTest.generates_query_metrics_report

4) 쿼리 한도(상한) 테스트​

거창한 성능 테스트는 아니고, 그냥 이런 규칙을 테스트로 걸어두는 방식이다.

  • “이 API는 쿼리 2개 이하여야 한다”
  • “이 이상이면 뭔가 이상하다”

이후 리팩토링 중 쿼리 수가 늘어나면 테스트가 바로 실패한다.

실제로 도움이 됐던 케이스​

/api/leaveRequest/pending에서 쿼리 수가 13까지 튀는 걸 리포트에서 발견했다.
원인은 DTO 변환 과정에서 user를 반복 조회하고 있었던 것.

@EntityGraph로 연관을 함께 로딩해 문제를 해결했고, 결과적으로 쿼리 수는 2로 줄었다.

정리​

  • 감으로 하던 튜닝을 숫자로 확인하게 됐다
  • Codex 도움으로 빠르게 구조를 잡고, 직접 맞게 수정하며 이해했다
  • report.md 파일을 생성하여 결과를 쉽게 확인할 수 있다.
